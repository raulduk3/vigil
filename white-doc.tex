{\bfseries\large Vigil: Provable Silence Tracking for Email Threads}

\textbf{Abstract.}
Vigil is a deterministic, event-sourced system for provable silence tracking in email communication. Users forward emails to a dedicated address; Vigil detects actionable requests, organizes messages into threads, and emits immutable events when silence exceeds configured thresholds. The system produces replayable audit trails proving what was observed and when. Vigil never accesses inboxes, owns deadlines, creates reminders, infers urgency, or automates responses. All behavior is explainable by replaying immutable events.

\textbf{Problem.}
Email is where obligations live, yet email clients are designed for reading and replying—not for tracking which conversations await response. Quiet failures occur when requests go unanswered and no one notices the silence. Vigil provides \textbf{confidence through observation}: the assurance that silence is not going unnoticed.

\textbf{Product Capability.}
Vigil delivers one and only one capability: \textbf{provable silence tracking for email threads}. The system:
\begin{itemize}
\item Organizes forwarded emails into conversation threads
\item Detects explicit actionable requests via bounded LLM extraction
\item Tracks elapsed time since last thread activity
\item Emits alerts when silence crosses policy thresholds
\item Produces immutable, replayable evidence timelines
\end{itemize}

\textbf{Non-Goals.}
Vigil does not own deadlines, create reminders or tasks, infer urgency or importance, tell users what to do, or automate responses. Humans retain full responsibility.

\textbf{Core Abstraction: Watchers.}
A \emph{watcher} is an isolated monitoring scope with a unique ingestion email address (e.g., \texttt{finance-a7f3k9@vigil.run}) and policy configuration. Each watcher owns its threads, alerts, and notification settings. Users forward relevant emails to create tracked conversations.

\textbf{Thread Model.}
Threads represent communication context only—not obligations. A thread opens when \texttt{ACTION\_REQUEST\_OBSERVED} is emitted, updates activity on every \texttt{THREAD\_EMAIL\_ADDED}, and closes only via explicit closure signal or user action. Threads do not own deadlines and cannot auto-close from time or silence.

\textbf{Bounded LLM Extraction.}
The LLM answers exactly one question: \emph{Does this message contain an explicit request that reasonably expects a response?} If yes, emit \texttt{ACTION\_REQUEST\_OBSERVED} with the minimal verbatim source span and request type (confirmation, approval, response, review, or unknown). The LLM never extracts deadlines, infers urgency, or influences control flow.

\textbf{Silence Tracking.}
On each \texttt{TIME\_TICK} (every 15 minutes), the runtime computes silence duration for open threads. When \texttt{hours\_since\_activity} crosses \texttt{silence\_threshold\_hours}, an immutable \texttt{SILENCE\_THRESHOLD\_EXCEEDED} event is emitted and an alert is queued. Alerts fire only on threshold transitions, never repeatedly.

\textbf{Event-Sourced Architecture.}
The PostgreSQL event store is the sole source of truth. Events are append-only and never modified. State is always derived by replaying events. Active event types:
\begin{itemize}
\item \texttt{EMAIL\_RECEIVED}, \texttt{THREAD\_EMAIL\_ADDED}
\item \texttt{THREAD\_OPENED}, \texttt{THREAD\_CLOSED}
\item \texttt{ACTION\_REQUEST\_OBSERVED}, \texttt{CLOSURE\_SIGNAL\_OBSERVED}
\item \texttt{SILENCE\_THRESHOLD\_EXCEEDED}
\item \texttt{ALERT\_QUEUED}, \texttt{ALERT\_SENT}
\item \texttt{TIME\_TICK}, \texttt{WATCHER\_*}, \texttt{POLICY\_UPDATED}
\end{itemize}
Legacy events (\texttt{HARD\_DEADLINE\_EXTRACTED}, \texttt{REMINDER\_*}, etc.) remain replayable for backward compatibility but produce no runtime behavior.

\textbf{Policy Model.}
\texttt{WatcherPolicy} contains: \texttt{allowed\_senders}, \texttt{silence\_threshold\_hours}, and \texttt{notification\_channels}. No deadline or urgency fields.

\textbf{Infrastructure.}
Backend: TypeScript/Bun with Hono. Event Store: PostgreSQL. LLM: OpenAI API (GPT-4o-mini). Inbound Email: Cloudflare Email Routing. Outbound Email: Resend API. Frontend: Next.js 14. Auth: JWT with Google/GitHub OAuth. Billing: Stripe (4 tiers).

\textbf{Security.}
Vigil never accesses inboxes. Only allowlisted senders create threads. PII is sanitized before storage. Email bodies are discarded after extraction. Rate limiting and HMAC-SHA256 webhook signatures prevent abuse.

\textbf{Business Model.}
Usage-based SaaS pricing with email ingestion volume as the primary billable unit. Tiers: Community (free, capped), Paid (per-email), Enterprise (contract with SLA and exports).

\textbf{Design Philosophy.}
Determinism over intelligence. Transparency over automation. Restraint over completeness. All behavior must be explainable by replaying events. The system can defend ``nothing happened'' with immutable evidence.
