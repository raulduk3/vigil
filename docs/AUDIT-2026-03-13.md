# Vigil Pre-Deploy Audit — 2026-03-13

Full security and consistency review before public deployment.

## Fixed (this commit)

### Critical

| Issue | Severity | Fix |
|---|---|---|
| **OAuth queries V1 schema** — `findOrCreateUser()` referenced `users` table and `accounts.account_id`/`accounts.owner_email` which don't exist in V2. Google/GitHub OAuth login would crash on any attempt. | Critical | Rewrote to use V2 `accounts` table (id, email, oauth_provider) |
| **Hardcoded fallback secrets** — `thread-actions.ts` fell back to `"vigil-dev-secret"`, webhook signing fell back to `"vigil-webhook-secret"` when env vars missing. In production with missing config, action tokens and webhooks would use known secrets. | Critical | Removed fallbacks. Missing JWT_SECRET now throws. Missing WEBHOOK_SIGNING_SECRET returns "unsigned" with warning. |
| **Hono CVEs** — v4.11.3 had JWT algorithm confusion (GHSA-3vhc, GHSA-f67f), arbitrary file access (GHSA-q5qw), and 7 other advisories. | High | Bumped to 4.12.7. All Hono CVEs resolved. |
| **No rate limiting** — Login, registration, ingestion, and all API endpoints had zero rate limiting. Brute force, credential stuffing, and ingestion abuse all trivially possible. | High | Added in-memory sliding window rate limiter. Auth: 10/15min. Ingest: 100/min. API: 120/min. Headers: X-RateLimit-Limit, Remaining, Reset, Retry-After. |
| **No email validation** — Registration accepted any string as email. | Medium | Added regex + length validation (254 char max). |
| **Bogus npm package** — Frontend had `"nextjs": "^0.0.3"` (not Next.js, a placeholder package). Supply chain risk. | Medium | Removed. |
| **Frontend vulnerabilities** — Next.js and rollup had 5 high/moderate CVEs. | Medium | `npm audit fix` resolved all. 0 vulnerabilities remaining. |

## Known Issues (not yet fixed)

### Must Fix Before Launch

| Issue | Severity | Notes |
|---|---|---|
| **SERVER_INSTANCE_ID invalidates all tokens on restart** | High | JWT access tokens include a `instance_id` check against a `crypto.randomUUID()` generated at boot. Every server restart logs out every user. For MVP this is acceptable but must be fixed before real users. Options: (a) persist instance ID, (b) remove instance check and rely on token expiry, (c) use RSA key pair. |
| **No usage metering** | High | No billing infrastructure exists. Invocations aren't counted per account per billing period. Need: `usage` table (account_id, period, email_count, cost_usd), increment on each invocation, expose via API and dashboard. |
| **No watcher limit enforcement** | Medium | `accounts.plan` column exists but nothing enforces limits. Free tier (PRODUCT.md): 1 watcher, 50 emails/month. Need middleware that checks plan limits on watcher creation and email ingestion. |
| **No Stripe integration** | Medium | Stripe packages in frontend (`@stripe/react-stripe-js`, `@stripe/stripe-js`) but no backend integration. Need: metered subscription creation, usage reporting to Stripe, billing portal. |
| **Rate limiter is in-memory** | Medium | Fine for single-process deploy. Won't work behind a load balancer. Replace with SQLite-backed or Redis-backed limiter before scaling past one server. |
| **No CSRF protection** | Medium | Token-based auth mitigates most CSRF but OAuth callback flow and cookie-based sessions (if added) need protection. |
| **No password reset flow** | Low | Frontend has `/auth/forgot-password` and `/auth/reset-password` pages but no backend implementation. |
| **OAuth state parameter not persisted** | Low | `oauthStart` generates a state UUID but doesn't store or verify it in the callback. CSRF vector on OAuth flow. |

### Architecture Notes

| Item | Status |
|---|---|
| Email bodies never stored | ✅ Confirmed. SHA-256 hash only. Body processed in memory. |
| SQLite WAL mode | ✅ Enabled on init. |
| Foreign keys | ✅ Enabled on init. |
| FTS5 sync triggers | ✅ Insert/update/delete triggers keep index current. Rebuild on startup. |
| Alert budgeting | ✅ 5 alerts/24h per watcher. Suppressed alerts noted for digest. |
| Thread cascade delete | ✅ `ON DELETE CASCADE` for memories→threads. |
| Webhook HMAC signing | ✅ SHA-256, but requires WEBHOOK_SIGNING_SECRET to be set. |
| Thread action tokens | ✅ HMAC-signed, 7-day expiry, base64url encoded. |
| Password hashing | ✅ `Bun.password.hash()` (argon2id by default). |
| Refresh token rotation | ✅ Old token revoked on each refresh. |

## Deploy Readiness

**Backend: ready for staging deploy.** Core functionality (auth, ingestion, agent, alerts) is solid. Security issues fixed. Rate limiting in place.

**Frontend: functional but outdated.** Current dashboard works but doesn't match PRODUCT.md three-panel design. Landing page references old tiered pricing. Usable for testing, not for public launch.

**Billing: not ready.** No metering, no Stripe integration, no usage enforcement. Must be built before accepting payments.

**Recommended deploy sequence:**
1. Deploy backend to VPS/fly.io (staging)
2. Point api.vigil.run DNS
3. Test real email flow end to end
4. Build usage metering + plan enforcement
5. Frontend redesign
6. Stripe integration
7. Public launch
